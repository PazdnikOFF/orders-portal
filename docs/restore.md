# Восстановление из резервной копии

Восстановление выполняется **на лету**, без остановки портала, командой
`pg_restore --clean` в существующую базу.

> ⚠️ Восстановление заменяет текущее состояние БД состоянием на момент копии.
> Изменения, сделанные после создания копии, будут потеряны. Перед
> восстановлением рекомендуется создать свежую копию.

## Способ 1. Через интерфейс (рекомендуется)

1. Войдите администратором → меню **Резервные копии**.
2. Найдите нужную копию по дате/времени.
3. Нажмите **Восстановить**, подтвердите действие.
4. Дождитесь сообщения об успехе. Операция записывается в журнал действий.

## Способ 2. Из консоли

```bash
# список копий
docker compose exec web ls -la /data/backups

# восстановление конкретного файла
docker compose exec web python manage.py shell -c \
  "from apps.backups.models import Backup; from apps.backups.services import run_restore; \
   b=Backup.objects.get(filename='portal-YYYYMMDD-HHMMSS.dump'); print(run_restore(b))"
```

Либо напрямую через `pg_restore`:

```bash
docker compose exec -T db pg_restore --clean --if-exists --no-owner \
  -U portal -d portal < /path/to/portal.dump
```

## После восстановления

- Каталог копий (`backups_backup`) приводится к состоянию на момент копии —
  файлы копий на диске при этом сохраняются и при необходимости могут быть
  пересозданы (повторный «Создать копию сейчас»).
- Проверьте доступность портала и корректность данных.
- При расхождении файлов заказов с БД проверьте том `order_files`.

## Полное аварийное восстановление сервера

1. Установите Docker и разверните проект (см. [deployment.md](deployment.md)).
2. Восстановите тома `order_files` (файлы заказов) и `backups` из внешнего
   хранилища.
3. Запустите стек: `docker compose up -d`.
4. Восстановите БД из последней копии (способы выше).
